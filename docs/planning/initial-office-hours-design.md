# Design: Notion to BEO Document MVP Pipeline

## Problem Statement
Build a narrow, shippable MVP that turns structured event records into an operational BEO-style document with minimal moving parts. The user-defined scope is explicit: Notion as the source of truth, n8n polling trigger, one generated document type, flat Google Drive destination, one notification email, and status/error writeback to Notion.

## Demand Evidence
Evidence provided in this session is implementation-first scope discipline, not external customer proof yet. Signal quality is still good because the scope deliberately cuts complexity:
- "One document type generated"
- "Flat folder"
- "Single email notification"
- "Complete / Error status writeback"

Evidence gap to close next: one real operator confirming this replaces a current manual workflow step they do every event.

## Status Quo
Likely current workflow is semi-manual operations coordination:
- Event details tracked in a tool (or ad hoc notes)
- Staff manually assembling prep/BEO docs
- Manual upload to Drive and manual notification
- Manual status updates after completion/failure

That status quo is error-prone around handoffs and creates invisible failures.

## Target User & Narrowest Wedge
Target user: event operations coordinator or banquet manager preparing event packets.

Narrowest wedge for this week:
- One Notion database row with required fields
- One automation trigger every 10 minutes
- One output artifact (start with prep list or BEO)
- One destination (flat Drive folder)
- One notification recipient
- One status contract in Notion (`Queued` -> `Complete` or `Error`)

## Constraints
- Keep orchestration in n8n
- Keep source data in Notion
- Do not build three document types in MVP
- Avoid nested Drive folder strategy until pain is real
- Keep notifications single-recipient initially
- Must write operational status back to Notion

## Premises
1. A 10-minute polling cadence is acceptable for operations latency and does not break user expectations.
2. A single generated document type is enough to validate workflow value before broadening output scope.
3. Drive flat-folder sprawl is manageable at MVP scale and should be delayed until concrete retrieval pain appears.
4. The right success bar is reliability and traceability (`Complete`/`Error` writeback), not feature breadth.

## Approaches Considered
### Approach A: Google Docs Template Fill via n8n (Minimal Viable)
Summary: Use n8n to poll Notion, map fields into a Google Docs template, export or keep native Doc, upload to one Drive folder, email link, update Notion status.
Effort: S
Risk: Low
Pros:
- Fastest path with familiar tools and low glue code
- Native Docs collaboration and easy manual override
- Smallest integration surface for MVP debugging
Cons:
- Template logic is less expressive for advanced conditional formatting
- May require tighter template discipline as fields grow
Reuses:
- Existing `beo-automation` BEO field model patterns (`lib/types.ts`)
- Existing workflow/status semantics already present in UI/API concepts

### Approach B: Carbone Document Generation via n8n (Ideal Architecture)
Summary: Use Carbone templating for generation quality/format control, then upload output to Drive and write back status.
Effort: M
Risk: Med
Pros:
- Better control over complex document layouts and conditional sections
- Easier long-term path to multiple output formats
Cons:
- Adds one more external dependency and failure point in MVP
- Debugging template/data mismatch is slower in early rollout
Reuses:
- Same trigger/status pipeline, only generation engine swaps

### Approach C: Event-Driven Webhook Pipeline (Creative/Lateral)
Summary: Replace polling with event-driven updates (Notion webhook proxy), queue jobs, generate document asynchronously.
Effort: L
Risk: High
Pros:
- Lower latency and cleaner long-term scaling model
- Better foundation for multi-document orchestration later
Cons:
- Overbuild risk for MVP and higher setup/ops complexity now
- More infrastructure before first validated value
Reuses:
- Limited reuse from current setup, mostly net new pipeline plumbing

## Recommended Approach
Choose Approach A now because it compresses time-to-first-reliable-output with the fewest failure surfaces while preserving a clean migration path to Carbone if format complexity becomes the bottleneck.

## Open Questions
- Which single document type comes first: prep list or formal BEO?
- Which Notion status values will be authoritative (exact enum list)?
- What is the retry policy before marking `Error`?
- Should the email body include summary metadata (event date/client) or link-only?
- What minimum required Notion fields must validate before generation starts?

## Success Criteria
- 100% of eligible test events move from queued to terminal status (`Complete` or `Error`) with no silent failures.
- Generated doc link appears in Notion within one automation cycle after trigger (<= 10 minutes + generation time).
- Notification email is sent exactly once per successful run.
- On failure, Notion stores machine-readable error reason and timestamp.
- Manual recovery path exists: fixing source data and rerunning transitions to `Complete` without manual DB surgery.

## Distribution Plan
This MVP is an internal workflow service, not an end-user binary/package.
Distribution channel:
- n8n workflow(s) deployed in your current automation environment
- Google Docs template maintained in shared Drive
- Output artifacts stored in one shared Drive folder

CI/CD:
- Initial phase can be manual workflow versioning with export snapshots
- Add automated workflow promotion later if failure rate or change volume rises

## Dependencies
- Notion integration/token with DB read/write scope
- n8n instance with Notion + Google integrations configured
- Google Docs template and Drive folder permissions
- SMTP/Gmail connector for single-recipient notification
- Finalized field mapping contract from Notion -> document placeholders

## The Assignment
Run a 3-event pilot this week using real event records. For each event, log: trigger time, doc generation time, completion/error status, and whether a human had to intervene. Do not add scope until these 3 runs are clean.

## What I noticed about how you think
- You explicitly constrained scope to MVP-safe primitives instead of chasing full automation breadth.
- You separated validation-critical work from nice-to-have structure (`flat folder now, hierarchy later`).
- You treated operational reliability (`Complete / Error writeback`) as first-class, which is the right backbone for automation systems.
- You pre-committed to single-recipient notification and one doc type, which reduces blast radius while proving value.
